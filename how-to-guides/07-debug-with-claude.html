Guide 07

How to Debug with Claude Code

How to describe problems so Claude can fix them — and how to build up a project memory of common errors so Claude gets better over time.

The Debugging Pattern

When something breaks, Claude needs three things from you:

  1. What you expected to happen
  2. What actually happened (the error or wrong behavior)
  3. Where it happened (which file, which action)
1

Copy the error message

If your terminal or browser shows an error, copy the whole thing. Don't paraphrase it.

Good:

claude
I'm getting this error when I click the submit button: TypeError: Cannot read properties of undefined (reading 'map') at RecipeList (src/components/RecipeList.tsx:14)

Bad:

claude
It's broken
⬚ Screenshot or screen recording

Browser console showing an error message being copied

2

Tell Claude what you were doing

Add context about when the error happens:

claude
This error shows up when I click "Save Recipe" after filling out the form. It was working before I added the ingredients field.
3

Point to the file

If you know which file is involved:

claude
The error is in src/components/RecipeList.tsx. Can you take a look?

Claude Code can read your files directly, so pointing it to the right place is enough.

4

Let Claude investigate

After giving context, Claude will read the relevant files and suggest a fix. Review what it proposes before accepting.

Tip

If Claude's fix doesn't work, say "that didn't fix it, here's what's still happening:" and share the new error. Don't just say "still broken."

Teach Claude Your Common Errors

Once you've fixed a bug, you can make sure Claude never hits the same issue blind again. Add it to your CLAUDE.md:

CLAUDE.md
## Known Issues and Gotchas - Instant DB `useQuery` returns `undefined` on first render -- always check for `isLoading` before accessing data - Tailwind classes don't apply in new components unless the file is in the `content` array in `tailwind.config.js` - Environment variables must be prefixed with `VITE_` to be accessible in the frontend

Over time this becomes a project-specific debugging reference. At the start of each session, Claude reads it and won't make the same mistakes twice.

You can also ask Claude to log it for you:

claude
That fixed it. Add a note to CLAUDE.md so we don't hit this again.

Claude will write the gotcha directly into your CLAUDE.md.

The Template

When in doubt, use this format:

claude
I'm working on [feature/page]. When I [action], I expected [expected behavior]. Instead, I get [actual behavior/error]. The relevant file is [file path].

Common Debugging Scenarios

You see... Tell Claude...
White screen "My app shows a blank white page. Check the browser console for errors."
Wrong data showing "The recipe list shows the wrong recipes. Check how data is fetched in [file]."
Styling looks off "The layout is broken on mobile. Check the Tailwind classes in [component]."
Build fails Copy the full build error and paste it.

Troubleshooting

Claude fixes one thing but breaks another

Commit working code before each fix so you can roll back.

The error message is confusing

Just paste it as-is. Claude is good at reading error messages even when you can't.

Nothing shows up in the terminal

Check your browser's developer console (right-click → Inspect → Console tab).

Same error keeps coming back

That's a sign it belongs in CLAUDE.md. Ask Claude to add it as a known issue.
