Guide 03

How to Write a Good CLAUDE.md

Create a CLAUDE.md that gives Claude Code the context it needs — and verify it's actually working.

What Is CLAUDE.md?

It's a file you put in your project's root folder. Claude Code reads it at the start of every session. Think of it as a briefing document — it tells Claude what this project is, how it's structured, and how you want it to behave.

Prerequisites

  • Claude Code installed and running — see Guide 01 if you haven't done this yet
  • A project folder open in your terminal
1

Let Claude Generate a Starting Point

Before writing one from scratch, try this. Start Claude Code in your project folder and say:

claude
Look at this project and create a CLAUDE.md for me based on what you see.

Claude will read your files and generate a starting point. You can edit from there — much faster than writing it cold.

Note

Claude Code may also have a built-in /init command that does this automatically. Try it and see if it works in your version.

2

Add the Three Core Sections

A good CLAUDE.md has three things: Identity, Rules, and Context.

Identity — What is this project? One or two sentences.

CLAUDE.md
# My Recipe App A web app that helps people find recipes based on ingredients they have at home. Built with React and Instant DB.

Rules — How should Claude behave in this project?

CLAUDE.md
## Rules - Keep the UI simple and mobile-friendly - Use Tailwind CSS for all styling - Never delete existing recipes from the database without asking first

Context — What does Claude need to know about the structure?

CLAUDE.md
## Key Files - `src/App.tsx` -- Main app component - `src/db.ts` -- Database connection and queries - `instant.schema.ts` -- Database schema
3

Keep It Short — Use References Instead

Your CLAUDE.md doesn't need to contain everything. If some context only matters for one part of the project, put it in a separate file and reference it:

CLAUDE.md
## Context See @docs/architecture.md for how the data layer works. See @src/components/CLAUDE.md for component conventions.

The @filename syntax tells Claude to load that file too. This way your main CLAUDE.md stays lean and Claude only loads deep context when it's relevant.

Tip

Sub-folders can have their own CLAUDE.md too. Put component-specific rules in src/components/CLAUDE.md and they'll load when Claude is working in that folder.

4

Verify Claude Actually Read It

Don't assume Claude loaded your CLAUDE.md. At the start of a session, ask:

claude
What does my CLAUDE.md say?

If Claude can summarize it accurately, it's loaded. If it hesitates or gets it wrong, your file might be in the wrong location or have a syntax issue.

5

Keep It Updated

Every time you find yourself correcting Claude about the same thing, add it to CLAUDE.md. You can also just tell Claude:

claude
Add a rule to CLAUDE.md: always use early returns instead of nested if statements

It will update the file for you.

Bad: "This is a project. Please help me."

Good: "A personal budget tracker built with React. Use Tailwind for styling. The main data lives in src/data/transactions.ts. Never mock data — always use the real Instant DB."

Troubleshooting

Claude keeps forgetting my preferences

Make sure your CLAUDE.md is in the project root (same level as package.json), not inside a subfolder.

Claude ignores a rule

Be more specific. "Write clean code" is vague. "Use early returns instead of nested if statements" is actionable.

File is getting too long

If it's over 50 lines, use @references to split context into separate files. Keep the main CLAUDE.md to rules and structure only.

Not sure it loaded

Ask Claude "What does my CLAUDE.md say?" to confirm.
