Guide 11

How to Write a Spec Before You Build

How to write a one-page brief that makes Claude Code dramatically more effective — and saves you from rebuilding the same thing three times.

Why This Matters

The most common mistake in vibe coding is starting with "build me X." Claude will build something, but it probably won't be what you imagined. You'll spend more time correcting than building.

Five minutes of writing before you code saves hours of redirecting.

The One-Page Spec

Write answers to these four questions before you open Claude Code:

1

What is this? (1-2 sentences)

Describe the thing in plain English. Pretend you're explaining it to a friend.

Example

"A web app where users can log their daily mood and see a chart of patterns over time."

Not: "A mood tracking app" (too vague) or three paragraphs about the vision (too much).

2

Who uses it and what do they do?

One user, one action. The main thing.

Example

"Someone opens the app, picks their mood from 5 options, and adds a one-line note. That's it."

This forces you to define your core feature before adding anything else.

3

What does it look like? (rough sketch is fine)

List the screens or pages. Don't design them - just name them.

Example
  • Home screen: mood picker + text box + submit button
  • History screen: list of past entries
  • Chart screen: mood over the last 30 days
4

What are the rules?

Any constraints Claude needs to know upfront:

Example
  • Mobile-first - most users are on their phone
  • No user accounts for now - data stays local
  • Use Tailwind for styling
  • Keep it simple - no animations, no fancy transitions

How to Use It

Once you have your spec, paste it into Claude Code as your first message:

claude
Here's what I want to build: [paste your spec] Let's start with the home screen. What's the first step?

Claude will read the whole spec and make decisions consistent with it throughout the session.

Tip

Save your spec as SPEC.md in your project folder and reference it in your CLAUDE.md. Claude will re-read it at the start of every session.

What a Good Spec Looks Like

Bad Spec

"Build a recipe app with search and user accounts and a meal planner and grocery lists."

Good Spec

"A recipe search app. Users can search recipes by ingredient. There's a results list and a detail page per recipe. No user accounts yet. Built with React + Instant DB. Mobile-first, Tailwind for styling."

The good spec has a clear scope, constraints, and a starting point. The bad spec is a feature list with no priority.

Troubleshooting

Not sure what you want

Write the bad version first. Give it to Claude and say "help me refine this spec before we start building." Claude is good at asking the right questions.

Spec keeps growing

Mark anything beyond the first version as "Layer 2." Get Layer 1 done first.

Claude is ignoring the spec

Put it in your CLAUDE.md so it loads every session. Or say "re-read the spec before continuing."
