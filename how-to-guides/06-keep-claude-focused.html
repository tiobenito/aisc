Guide 06

Context Management in Claude Code

Three Claude Code-specific habits that keep sessions clean and prevent Claude from getting confused.

Why Context Matters

Claude Code keeps a running memory of everything said in a session — every file it read, every change it made, every decision you discussed. This is powerful, but it can work against you: if you've been debugging one feature for an hour and then switch to building something new, Claude is still carrying all that old context. It can bleed into your new work in subtle ways.

These three habits prevent that.

1

Use /clear When Switching Topics

When you finish one task and start something unrelated, clear Claude's memory:

claude
/clear

This resets the session. Claude goes back to reading only your CLAUDE.md and the files you explicitly show it.

When to use it:

  • › You just fixed a bug and now want to build a new feature
  • › You finished the backend and are starting the frontend
  • › The conversation has gotten long and confusing
Tip

You don't need /clear between small related tasks. Use it when you're genuinely starting something new.

2

Use /compact for Long Sessions

If a session has been going for a while and Claude seems slower or less focused, try:

claude
/compact

This compresses the conversation history to save space. Claude keeps the important context but drops the noise. You stay in the session — unlike /clear, this doesn't wipe everything.

When to use it:

  • › You've been in the same session for 30+ minutes
  • › Claude starts repeating itself or giving off-track responses
  • › You get a warning about context getting full
3

Commit Before Experimenting

Before asking Claude to try something risky or unfamiliar — a big refactor, a new library, an architecture change — commit your current working state:

zsh
$ git add . $ git commit -m "working state before [experiment]"

Or just ask Claude:

claude
Commit the current state before we try this

This gives you a safe save point. If the experiment goes wrong, you can always go back. See guide 05 for how to recover.

The Pattern

For any significant work session:

  • › Start fresh with /clear (or just a new terminal)
  • › Work on one thing at a time
  • › Commit when it works
  • › /clear before switching to something new

Troubleshooting

Claude seems confused about which file we're working on

/clear and re-state what you're doing.

Claude keeps referencing something from earlier in the session

That's context bleed. /clear fixes it.

Session feels sluggish

Try /compact first. If that doesn't help, /clear and start fresh.
