Guide 15

Understanding AI Models

What models are, how they're different, and when to switch between them — so you stop guessing and start choosing.

What Is a Model?

Think of it like a car engine. The app you use — Claude Code, ChatGPT, Cursor — is the car. The model running underneath is the engine. Same car can have different engines. Same interface can run different models.

When someone says "I used Claude," they probably mean the Claude app. But the thing that actually generates the response is a model — Sonnet, Opus, Haiku, etc. This distinction matters because the model you're using affects the quality, speed, and cost of every response you get.

You don't need to understand how models are built. You just need to know which one to pick for the job.

The Claude Family

This is what we use in the course. Three tiers, each built for a different kind of work:

Model Speed Cost Best For
Haiku Fastest Cheapest Quick questions, simple formatting, autocomplete, batch tasks
Sonnet Fast Moderate Most coding, writing, analysis — the everyday workhorse
Opus Slower Most expensive Complex reasoning, architecture decisions, subtle bugs
Key Idea

These aren't "good, better, best." They're different tools for different jobs. Using Opus to rename a variable is like hiring a lawyer to proofread a grocery list. Sonnet is what Claude Code uses by default — and it's the right choice for most of your work.

OpenAI Models (Brief Overview)

You'll encounter these outside the course. Here's where they roughly map:

Model Comparable To Notes
GPT-4o Sonnet OpenAI's general-purpose model. Good all-rounder.
o1 / o3 Opus + extended thinking OpenAI's "reasoning" models. They think longer before answering.

For this course, you don't need to worry about OpenAI models. Just know they exist and roughly where they sit.

When to Change Your Model

Default to Sonnet for 90% of your work. It's fast, capable, and cost-effective. Most tasks don't need anything more.

Switch to Opus When:

  • › You're debugging something subtle that Sonnet can't figure out
  • › You're planning the architecture for a new feature or project
  • › The task requires complex, multi-step reasoning
  • › You need nuanced judgment (e.g., "should I restructure this or just patch it?")
  • › Sonnet gave you a wrong or shallow answer and you want a stronger second opinion

Stick with Haiku For:

  • › Quick lookups or simple questions ("what does this error mean?")
  • › Simple formatting or text transformations
  • › Batch operations where you're running many small tasks and speed matters
  • › Autocomplete suggestions while coding

Thinking Modes (Extended Thinking)

Some models can "think longer" before answering. In Claude Code, this shows up as extended thinking — you'll see Claude working through the problem step by step before giving you a response.

The model spends more time reasoning through the problem internally. This produces better answers for hard problems, but takes longer and costs more.

When to Use Extended Thinking

  • › Complex debugging where the bug isn't obvious
  • › Architectural decisions with multiple tradeoffs
  • › Tricky logic or multi-step problems
  • › When you get a wrong answer in normal mode and want Claude to think harder

When NOT to Use Extended Thinking

  • › Simple tasks (drafting a message, renaming files, basic questions)
  • › Anything where speed matters more than depth
  • › Tasks Sonnet handles fine on the first try
In Claude Code

You can nudge Claude toward deeper thinking by using phrases like "think carefully about this" or "think step by step" in your prompt. Claude Code also has explicit extended thinking controls — when enabled, you'll see the thinking process in the output.

Cost Awareness

More capable = more expensive = slower. That's the tradeoff.

Haiku Sonnet Opus
Relative cost $ $$ $$$$
Relative speed Fastest Fast Slower
Quality ceiling Good for simple tasks Great for most tasks Best for hard tasks
Budget Tip

For students on free tiers or limited budgets, Sonnet is the right default. Don't upgrade to Opus unless Sonnet is genuinely failing at the task. The mistake to avoid: using the most powerful model for everything "just in case." You'll burn through your budget faster, get slower responses, and the output won't actually be better for simple tasks.

The Decision in Practice

When you sit down to work, ask yourself one question: is this task simple, normal, or hard?

1

Simple Task

Quick question, formatting, lookup.

Use: Haiku or Sonnet, no extended thinking.

2

Normal Task

Coding a feature, writing, analysis.

Use: Sonnet, no extended thinking. This is 90% of your work.

3

Hard Task

Subtle bug, architecture decision, complex reasoning.

Use: Sonnet first. If it struggles, switch to Opus or enable extended thinking.

The Framework

Start with the lighter option and escalate only when needed. That's it. You don't need to memorize model specs or pricing tables. Just ask: is Sonnet handling this well? If yes, keep going. If not, try harder.
